# ТЗ: BikeSouth bike rental demand regression

## Цель

Построить воспроизводимое ML-решение для прогноза почасового спроса на
велосипеды (`Rented Bike Count`) по погодным, календарным и режимным признакам.
Решение должно показать, можно ли улучшить предоставленную компанией линейную
baseline-модель за счет нелинейных моделей `KNeighborsRegressor` и
`DecisionTreeRegressor`.

## Обязательные входные файлы

- `data/raw/ds_s14_train_data.csv` - train-выборка для EDA, обучения,
  кросс-валидации и подбора гиперпараметров.
- `data/raw/ds_s14_test_data.csv` - финальная test-выборка. Не используется
  для подбора модели, признаков или гиперпараметров.
- `models/baseline_linear_regression_pipeline.joblib` - предоставленный
  baseline pipeline линейной регрессии.

## Критерии приемки

- В ноутбуке сохранены исходные ячейки задания/ревьюера; авторское решение
  добавляется после раздела `# **Авторское решение**`.
- Baseline-модель загружена, ее структура показана, качество оценено на train и
  test по `RMSE`, `MAE`, `R2`.
- Для новых моделей test-выборка используется только один раз после выбора
  лучшего подхода по train CV.
- Обучены минимум две обязательные нелинейные модели:
  `KNeighborsRegressor` и `DecisionTreeRegressor`.
- Для обеих моделей подбор гиперпараметров выполнен через Optuna.
- Для обеих моделей используется единая 5-fold cross-validation схема.
- Основная метрика оптимизации - `RMSE`; дополнительные метрики - `MAE`, `R2`.
- Все обучаемые преобразования находятся внутри `sklearn.pipeline.Pipeline`.
- Проведен EDA с выводами, которые явно связаны с решениями по
  предобработке/моделированию.
- Подготовлен кастомный sklearn-compatible transformer для дополнительных
  признаков, если он улучшает или методологически усиливает решение.
- Финальная модель сравнена с baseline на одной и той же test-выборке.
- Итоговый pipeline сохранен и сразу проверен через `joblib.load`.
- Вместе с pipeline сохранены metadata, model card, manifest компонентов,
  test predictions и inventory артефактов.
- Код кастомного feature engineering доступен как импортируемый Python-модуль,
  связан с model card/metadata через checksum и проверен на наличие обязательных
  имен, без которых сохраненный pipeline не загрузится.
- Финальный вывод написан как бизнес-рекомендация для заказчика, а не только
  как перечисление метрик.

## Стиль ноутбука

Ноутбук должен соответствовать уровню шаблонов `template*.ipynb`:

- введение с описанием заказчика, бизнес-боли, цели и критериев успеха;
- навигация/оглавление;
- методологическая рамка до кода;
- короткие, читаемые code cells;
- после каждого смыслового блока - подвывод;
- выводы пишутся живым языком без канцелярита: коротко, конкретно, с ответом
  на вопрос "что это значит для задачи";
- расчетные итоги с числами отделены от текстовой интерпретации;
- после EDA - таблица "наблюдение -> решение";
- в начале исследования задан единый словарь `technical_name ->
  description_ru`, который используется в графиках, таблицах и выводах;
- аккуратные графики: заголовок, подписи осей, единицы измерения,
  `plt.show()`, без служебных `<AxesSubplot...>`;
- на сравнительных графиках подписаны значения метрик и, где уместно, ключевые
  параметры моделей в технической и русской формулировке;
- непрерывные признаки визуализируются histogram/boxplot, дискретные и
  категориальные - countplot/barplot;
- таблицы не обрезают длинный текст: `pd.set_option("display.max_colwidth",
  None)`;
- финальный вывод отвечает на вопросы "что получилось", "за счет чего",
  "можно ли применять", "что проверить дальше".

## Методологические запреты

- Не использовать `ds_s14_test_data.csv` в Optuna, cross-validation, выборе
  признаков или выборе модели.
- Не делать `fit` imputers/scalers/encoders на всем train+test.
- Не заполнять пропуски глобальным `fillna` до pipeline.
- Не удалять выбросы только потому, что они видны на boxplot.
- Не сравнивать модели только по одной метрике без анализа устойчивости.
- Не сохранять модель без последующей проверки загрузки.
- Не сохранять `joblib` без кода всех кастомных функций/классов, которые нужны
  для инференса и повторной загрузки pipeline.
- Не оставлять графики без подписей и выводов.

## План реализации

1. Сохранить исходные ячейки задания и добавить авторское решение ниже.
2. Подготовить окружение, импорты, пути, `RANDOM_STATE`, версии библиотек.
3. Загрузить train/test/baseline, нормализовать схему колонок одной функцией.
4. Провести аудит данных: размеры, типы, пропуски, дубли, диапазоны, target.
5. Зафиксировать словарь технических признаков и русских описаний.
6. Оценить baseline train/test и зафиксировать его ограничения.
7. Провести EDA target, погодных, категориальных и time-period признаков.
8. Сформировать таблицу EDA-решений для pipeline и feature engineering.
9. Создать `BikeFeatureEngineer` и фабрики pipeline для KNN/tree.
10. Провести `DummyRegressor` sanity check на train CV.
11. Настроить KNN и Decision Tree через Optuna с 5-fold CV.
12. Сравнить модели на train CV по `RMSE`, `MAE`, `R2`.
13. Выбрать лучшую модель, обучить на всем train, один раз оценить на test.
14. Провести интерпретацию: feature importances или permutation importance.
15. Сохранить pipeline, metadata, model card, manifest компонентов, inventory
    артефактов и test predictions; проверить reload.
16. Провести внутренний аудит качества, исправить замечания, перезапустить
    ноутбук сверху вниз.

## Внутренний контроль после каждого этапа

После каждого крупного этапа нужно проверить, но не выносить в ноутбук
служебный текст контроля:

- какие входы использованы;
- нет ли утечки из test;
- какие объекты созданы для следующего этапа;
- какие решения из предыдущего этапа переходят дальше;
- что будет считаться ошибкой на следующем шаге.
